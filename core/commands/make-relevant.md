# Make Relevant Command

Analyze this repository and adapt the harness to it. Produces `.claude/PROJECT.md` —
the repo-specific context file that every other harness command reads before doing
anything.

**Usage:** `/make-relevant [optional-focus-area]`

Run this **once after installing the harness**, and again whenever the repo's stack,
layout, or tooling changes materially.

## Why This Command Exists

The harness ships generic. Its planning commands say "focus on the relevant files"
and "run the validation commands" without knowing what those are in *this* repo. A
fresh agent booted into this project would have to rediscover the layout, the test
runner, and the conventions every single time — burning context before it does any
work.

This command does that discovery **once**, writes it down, and every other command
inherits it.

## CRITICAL: Where Output Goes

Write to `.claude/PROJECT.md`. **Do not edit the files in `.claude/commands/` or
`.claude/agents/`** — the harness installer overwrites those on every update, so any
edits there are silently lost. `PROJECT.md` is never overwritten by the installer.

## Instructions

### Step 1: Survey the Repository

Do not guess. Read the repo.

```bash
git ls-files | head -200          # actual tracked layout, not build artifacts
git ls-files | wc -l              # scale
```

Then read, in this order:
1. `README.md` — stated purpose and setup instructions
2. The manifest(s) — `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`,
   `Gemfile`, `pom.xml`, `Makefile`, `justfile`
3. CI config — `.github/workflows/*`, `.gitlab-ci.yml`, `Jenkinsfile`. **This is the
   most reliable source for the real validation commands**, because CI actually runs
   them.
4. Any existing `CLAUDE.md`, `CONTRIBUTING.md`, `docs/`

For a monorepo, identify each workspace and whether commands are run from the root
or per-package.

### Step 2: Verify the Commands Actually Run

**This step is mandatory and is the whole point of the command.**

For every candidate test / build / lint / typecheck command, **execute it** and record
what happened. A validation command that doesn't run is worse than no validation
command — it makes every downstream plan assert "zero regressions" while verifying
nothing.

For each command, record one of:
- ✅ **Verified** — ran, exited 0
- ⚠️ **Runs, currently failing** — ran, non-zero exit (record the failure; this is
  pre-existing repo state an agent must not mistake for its own regression)
- ❌ **Unavailable** — missing deps, needs credentials, needs a running service.
  Record *why*, and record the setup step needed.

If a command is slow (>2 min), note the runtime and whether a faster subset exists
(e.g. a single test file, `--watch=false`, an affected-only flag).

### Step 3: Determine the Relevant Files Map

Which directories does an agent actually need to touch to do work here, and which
should it ignore? Be specific and be exclusionary — the value is as much in what you
rule out as what you rule in.

Rule out: build output, vendored deps, lockfiles, generated code, fixtures, assets.

### Step 4: Extract Conventions From the Code, Not From Aspiration

Read 3-5 representative source files and 2-3 representative test files. Record what
the code *actually does*, not what a style guide wishes it did:

- Test framework, test file location (co-located vs `tests/`), naming pattern, and a
  real example of a typical test in this repo
- Error handling idiom (exceptions? result types? error returns?)
- Logging idiom, and whether errors surface anywhere an agent can read them
- Naming conventions for files, types, functions
- How new modules get wired in (barrel exports? registry? DI container? convention?)

### Step 5: Write `.claude/PROJECT.md`

Use this exact structure. Replace every `<placeholder>`. Omit a section only if it
genuinely does not apply, and say why.

### Step 6: Report

Tell the user:
1. The path written (`.claude/PROJECT.md`) and its section count
2. **Validation command status** — which are verified, which are broken, which are
   unavailable. Lead with this; it's the most actionable output.
3. Any **configuration drift** you found between `harness.config` values and reality,
   with the exact `harness.config` lines to change
4. Anything you could not determine and would need the user to confirm

## Rules

- **Never fabricate a command you did not run.** An unverified command in the
  Validation Commands table defeats the purpose of the table.
- **Do not modify the project's tracked source files.** This command is read-only
  with respect to the project; it writes exactly one file, `.claude/PROJECT.md`.
- **Do not commit or push.**
- If `.claude/PROJECT.md` already exists, read it first and update it — preserve any
  hand-written additions the user made, and note in your report what you changed.
- Prefer evidence over inference. "The CI workflow runs `X`" beats "the project
  probably uses `X`."
